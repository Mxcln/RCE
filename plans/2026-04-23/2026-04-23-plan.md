# RCE (Rubik's Cube Engine) 项目总计划

**日期：** 2026-04-23  
**语言：** Rust  
**工作区模式：** Cargo Workspace

## 目标

RCE 的长期目标是实现一个可扩展的 **3 阶魔方引擎**。  
但当前项目的 **首版目标** 明确收缩为：

- `rubiks-core`：完成稳定的状态引擎、扩展记法解析、朝向追踪与渲染数据输出
- `rubiks-cli`：完成基于 `rubiks-core` 的命令行工具与交互式 REPL

以下内容 **不属于首版交付范围**：

- 求解器实现
- OLL/PLL 公式目录与 case 识别
- 随机打乱生成器
- GUI / Web

`rubiks-alg` 与 `rubiks-solver` 在当前阶段只保留 **未来接口设计**，不要求进入可运行实现。

## 阶段划分

### Phase A：Core 引擎

目标：

- `CubeState`
- `validate()`
- move tables / `apply_move()`
- `MoveSequence`
- `ExtMoveSequence`
- `to_facelets()`
- `Orientation`
- `parse_notation()`
- `resolve_notation()`
- `Cube`

产出：

- 一个 canonical 状态引擎
- 一个面向用户交互的物理朝向包装层

### Phase B：CLI / REPL

目标：

- `rubiks new`
- `rubiks apply "<notation>"`
- `rubiks repl`
- ASCII 展开图渲染
- REPL 内置命令：`reset` / `history` / `show` / `validate` / `help`

产出：

- 用户可以在终端里输入扩展记法并实时查看魔方状态

### Phase C：Algorithm Catalog（预留）

当前只定义未来接口，不实现：

- `AlgEntry`
- OLL / PLL case 数据模型
- case-first 的 catalog 接口
- scramble 接口

### Phase D：Solver（预留）

当前只定义未来接口，不实现：

- `Solver` trait
- `Solution`
- `SolvePhase`
- `SolveError`

是否先做 LBL、reference solver，还是直接做 CFOP，留到 core 稳定后再决定。

## `rubiks-core`

### 职责

`rubiks-core` 是整个项目的基础，负责：

- cubie 层状态表示
- 基本面转
- 状态合法性校验
- 扩展记法解析
- 整魔方朝向追踪
- facelet 渲染数据输出

### 双层模型

`rubiks-core` 使用两个核心类型：

```text
CubeState                  Cube
领域核心状态引擎             面向用户交互的包装层
仅处理 canonical 操作        处理物理朝向与扩展记法
供未来 alg/solver 直接使用    供 CLI / REPL / GUI 使用
```

### `CubeState`

```rust
pub struct CubeState {
    corner_perm:   [u8; 8],
    corner_orient: [u8; 8],
    edge_perm:     [u8; 12],
    edge_orient:   [u8; 12],
}

pub struct CubeStateParts {
    pub corner_perm: [u8; 8],
    pub corner_orient: [u8; 8],
    pub edge_perm: [u8; 12],
    pub edge_orient: [u8; 12],
}
```

正式 API：

```rust
impl CubeState {
    pub fn solved() -> Self;
    pub fn try_from_parts(parts: CubeStateParts) -> Result<Self, CubeStateError>;
    pub fn parts(&self) -> CubeStateParts;
    pub fn is_solved(&self) -> bool;
    pub fn reset(&mut self);

    pub fn validate(&self) -> Result<(), CubeStateError>;
    pub fn is_valid(&self) -> bool;

    pub fn apply_move(&mut self, mv: Move);
    pub fn apply_sequence(&mut self, seq: &MoveSequence);

    pub fn to_facelets(&self) -> [[Color; 9]; 6];
}
```

`CubeState` 是整个引擎的 **领域核心数据结构**，只接受 **canonical frame** 下的基础面转。  
未来 `rubiks-alg` / `rubiks-solver` 都直接建立在它之上。

补充约束：

- `CubeState` 的内部数组保持私有
- 外部调用者通过 `CubeState::try_from_parts()` 做受控构造
- 如实现内部确实需要跳过校验，可提供 crate 内部的 unchecked 构造，但它不属于公共契约

### `Cube`

```rust
pub struct Cube {
    state: CubeState,
    orientation: Orientation,
}
```

正式 API：

```rust
impl Cube {
    pub fn solved() -> Self;
    pub fn from_state(state: CubeState) -> Self;
    pub fn from_state_with_orientation(
        state: CubeState,
        orientation: Orientation,
    ) -> Self;

    pub fn state(&self) -> &CubeState;
    pub fn orientation(&self) -> Orientation;
    pub fn into_state(self) -> CubeState;

    pub fn reset(&mut self);
    pub fn reset_orientation(&mut self);
    pub fn is_solved(&self) -> bool;
    pub fn validate(&self) -> Result<(), CubeStateError>;
    pub fn is_valid(&self) -> bool;

    pub fn apply_canonical_sequence(&mut self, seq: &MoveSequence);
    pub fn apply_ext_sequence(&mut self, seq: &ExtMoveSequence);
    pub fn apply_canonical_notation(&mut self, input: &str) -> Result<(), ParseError>;
    pub fn apply_notation(&mut self, input: &str) -> Result<(), ParseError>;
    pub fn to_facelets(&self) -> [[Color; 9]; 6];
}
```

约束：

- `Cube` 面向用户交互语义
- `Cube` 不是新的领域核心，只是 `CubeState + Orientation` 的受控包装
- `Cube` 不公开暴露内部字段，也不提供通用 `state_mut()`
- `Cube` 不暴露语义模糊的 `apply_sequence()`，而是显式区分 `apply_canonical_sequence()` 与 `apply_ext_sequence()`
- canonical `MoveSequence` 的执行只属于 `CubeState`
- `Cube::validate()` / `Cube::is_valid()` 只是委托给内部 `CubeState`
- `Cube::is_solved()` 只反映 cubie 状态，和 `orientation` 无关
- solver / alg 若需要访问核心状态，应直接依赖 `CubeState`，而不是通过公开字段间接操作 `Cube`
- `Cube::apply_canonical_sequence()` 直接委托给内部 `CubeState`，不读取也不更新 `orientation`
- `Cube::apply_ext_sequence()` 的内部流程是 `resolve_notation() -> 提交 canonical moves -> 更新 orientation`，它不返回错误
- `Cube::apply_canonical_notation()` 的内部流程是 `parse canonical notation -> apply_canonical_sequence()`
- `Cube::apply_notation()` 的内部流程是 `parse_notation() -> apply_ext_sequence()`，因此 `ParseError` 只来自文本解析阶段

### `Orientation`

`Orientation` 表示：

- `physical face -> canonical face`

它只负责：

- 处理 `x/y/z`
- remap 物理面转到 canonical 面转
- remap canonical facelets 到物理展开图

它不负责：

- 保存 cubie 状态
- 执行 move tables
- solver 坐标或剪枝

### 扩展记法解析

正式语义：

- `parse_notation()` 输入用户写下的扩展记法文本
- `parse_notation()` 输出保留原始语义的 `ExtMoveSequence`
- `resolve_notation()` 再结合当前 `Orientation` 输出 canonical frame 下的 `MoveSequence`
- `resolve_notation()` 同时返回新的 `Orientation`

API：

```rust
pub struct ExtMoveSequence(pub Vec<ExtMove>);

pub fn parse_notation(
    input: &str,
) -> Result<ExtMoveSequence, ParseError>;

pub struct ResolvedStep {
    pub input: ExtMove,
    pub canonical: Vec<Move>,
    pub orientation_before: Orientation,
    pub orientation_after: Orientation,
}

pub struct ResolvedSequence {
    pub source: ExtMoveSequence,
    pub steps: Vec<ResolvedStep>,
    pub flattened: MoveSequence,
    pub final_orientation: Orientation,
}

pub fn resolve_notation(
    seq: &ExtMoveSequence,
    orientation: Orientation,
) -> ResolvedSequence;
```

说明：

- `parse_notation()` 只负责解析，不处理朝向
- `resolve_notation()` 负责把扩展记法编译为 canonical 执行计划；对于合法的 `ExtMoveSequence` 它不返回错误
- UI / REPL / 公式展示应优先依赖 `ExtMoveSequence`
- solver / `CubeState` / 差分测试只依赖 `MoveSequence`

支持：

- `U D F B L R`
- `M S E`
- `x y z`
- `r l u d f b`
- 后缀 `'` 和 `2`

### 朝向表的来源

正式实现中，`tools/gen_orientation_tables.py` 是朝向表的唯一权威来源，它应生成：

- `crates/rubiks-core/src/orientation_tables.rs`

约束：

- `crates/rubiks-core/src/orientation_tables.rs` 是提交到仓库的生成文件
- 构建与测试直接依赖 `src/orientation_tables.rs`
- 不在 `cargo build` 过程中动态生成朝向表
- CI 应重新运行生成器，并检查 `orientation_tables.rs` 没有差异

## `rubiks-cli`

### 职责

`rubiks-cli` 是首版唯一用户入口，依赖 `rubiks-core`。

它负责：

- 展示还原状态或执行记法后的状态
- 提供 REPL
- 渲染 ASCII 展开图

### 子命令

首版只实现以下命令：

```text
rubiks new
rubiks apply "R U R' U'"
rubiks repl
```

说明：

- `new`：显示还原状态
- `apply`：从还原状态开始执行一段记法并渲染结果
- `repl`：进入交互模式

### REPL

REPL 使用 `Cube`，支持完整扩展记法。

内置命令：

- `reset`
- `history`
- `show`
- `validate`
- `help`

普通输入：

- 视为一段记法，执行后更新魔方状态并渲染

首版不提供：

- `solve`
- `scramble`
- `alg list`

这些命令在后续 `rubiks-alg` / `rubiks-solver` 真正实现后再接入。

## `rubiks-alg`（预留接口）

### 当前状态

不属于首版交付。

当前只保留未来设计原则：

- 使用 **case-first** 数据模型
- case 与 algorithm 分离
- 不从“算法文本”反推唯一 case 身份

### 未来数据模型方向

```rust

    pub case_id: String,

    pub algorithms: Vec<AlgEntry>,

pub struct PllCase {
    pub case_id: String,
    pub canonical_pattern: PllPattern,
    pub algorithms: Vec<AlgEntry>,

```

### 未来接口方向

```rust
pub struct CaseAlignment {
    // exact fields TBD during implementation

pub struct CaseMatch<'a, T> {
```

### 当前状态

## 测试策略

### `rubiks-core`

- `validate()` 覆盖非法状态
- 每个基本面转执行 4 次回到初始状态
- `apply_move(mv)` 后再应用 `mv.inverse()` 回到初始状态
- `parse_notation()` 保留扩展记法语义
- `resolve_notation()` 生成仅含基础面转的 canonical 序列
- `Cube::apply_notation()` 能正确更新 `orientation`
- `Cube::apply_notation("x")` 后 `is_solved() == true`
- `to_facelets()` 在 solved 状态下输出正确颜色
- orientation 表满足排列性和 24 朝向可达性

### `rubiks-cli`

- 纯文本 ASCII 输出行数正确
- `apply` 命令能正确渲染结果
- REPL 状态机可测试，不依赖真实 stdin
- `history` / `reset` / `validate` 行为正确

### 差分测试

如果条件允许，建议引入与 `ref/kociemba/` 的差分测试，用于验证：

- move tables
- facelet 映射
- 记法展开等价性
